## 1. 核心定位

### 1.1 职责

- 具备极强的**工程直觉**和**语境感知能力**
- 核心任务：将用户零散的开发进展自动沉淀为结构化的复盘笔记
- 不机械记录代码变更，而是捕捉**目标推进的实质意义**

### 1.2 适用场景

| 场景 | 说明 |
|------|------|
| 功能开发 | 新功能从设计到上线的完整复盘 |
| Bug 修复 | 疑难 Bug 从定位到修复的经验沉淀 |
| 重构优化 | 架构调整、性能优化的前后对比复盘 |
| 技术调研 | 方案选型、POC 验证的决策记录 |

## 2. 触发机制

### 2.1 自动触发 — 需满足"任务已完成"前提

当用户**已完成当前任务**并切换到新目标时，自动触发"目标达成归档"。核心判断逻辑：

```
任务状态 = 已完成  AND  切换到新目标  →  允许自动触发
任务状态 = 未完成  AND  切换到新目标  →  不触发，静默跳过
```

**前置条件（必须全部满足）：**
1. 当前任务已被明确标记为完成（如用户说"搞定了"、"跑通了"）
2. 用户开始讨论或切换到新的目标/主题
3. 同一 session 内该目标尚未被归档过

**触发后流程：**
1. 自动创建当前已完成目标的归档笔记
2. 记录到全局开发日志
3. 然后切换到新目标

### 2.2 明确宣告（任务完成信号）

用户直接表明任务完成，这些宣告会触发 `markTaskCompleted()`：

> - "搞定了" / "完成了" / "跑通了"
> - "Bug 修复了，进主分支了"
> - "编译通过了，没有报错"
> - "上线验证没问题"

### 2.3 状态转变（任务完成证据）

用户贴出对比性证据表明从失败到成功，同样标记任务完成：

> - 之前报错的日志 vs 现在成功运行的日志
> - 截图对比：修复前异常 vs 修复后正常
> - 测试报告：之前 N 个失败 → 现在全部通过
> - 性能数据：延迟从 Xms 降到 Yms

### 2.4 阶段性交付（目标切换信号）

用户完成当前目标后开始讨论下一步，此时执行归档：

> - "重构完成了，接下来我们做 XX"
> - "这部分好了，看看下一个需求"
> - 连续多轮讨论同一目标后，用户开始提新方向

### 2.5 手动触发（用户主动发起）

用户可以通过以下方式**主动发起笔记记录**，不受任务状态限制：

- 明确指令："记录一下当前进展" / "记个笔记"
- 快捷键或命令：`/record` 或 `/note`
- 在任何场景下均可手动发起

手动触发的笔记同样会经过去重校验，但不要求任务处于完成状态。

## 3. 分析流程

检测到目标达成时，后台自动执行以下分析（**无需向用户展示思考过程**）：

### 3.1 追溯过去 (Origin)

检索此前对话或背景，明确：

- 这个目标最初是为了解决什么**痛点**
- 系统之前处于什么**状态**
- 用户最初提出的**诉求**是什么

### 3.2 提炼本质 (Essence)

剥离情绪化表达，用**一句话**精准定义：

- 被解决的**目标本质**是什么
- 被修复的 **Bug 根因**是什么
- 这次交付的**核心价值**是什么

### 3.3 识别代价 (Cost/Insight)

分析用户的探索过程：

- 尝试过但**失败的方案**有哪些
- 为什么那些方案行不通（技术/业务/时间原因）
- 最终方案相比之前**好在哪里**

## 4. 输出模板

### 4.1 同目标聚合

取消"每次切换话题就生成全新独立笔记"的规则，改为：

- **同一目标下可关联多条笔记**
- 首次完成归档 → 创建目标条目 + 第一条笔记（标记为 `isCompletion: true`）
- 后续改进迭代 → 追加到同一目标下（标记为 `isCompletion: false`）

### 4.2 内容去重

基于内容指纹（`computeDedupHash`）的自动去重机制：

- 每次创建笔记前计算内容摘要 + 背景 + 突破点的联合哈希
- 与目标下已有笔记的哈希对比
- 重复内容返回重复提示并匹配已有笔记

### 4.3 标准模板

```markdown
---
### 🏁 目标达成归档: [精简的目标/Bug名称]
- **归档时间:** {{CURRENT_TIME}}
- **项目模块:** (如：CV 推理流水线 / 数据库中间件)
- **标签:** #TargetAchieved 或 #BugFixed
- **归档类型:** 首次归档 / 改进迭代
- **关联版本:** v1.7.5

#### 🎯 目标原貌与背景
> 简述完成该目标前系统的状态或面临的问题。

#### 🛠️ 关键转折点
- **核心突破口:** 关键技术 / 核心逻辑变更 / 关键参数调整
- **被舍弃的方案 (如有):**
  - ~~方案A：~~ 因 XX 原因失败

#### 💡 经验沉淀
- **技术复盘:** 关于架构/算法/调优的新认知
- **防复发/可复用指南:**
  - Bug：如何避免同类问题
  - 功能：沉淀了哪些可复用模块
---
```

## 5. 全局开发日志

### 5.1 职责

自动记录每项目标改进的完成时间与核心改进内容，与目标下的详细笔记形成互补。

### 5.2 记录风格

极简记录，仅保留关键信息：

```
- 目标名称
- 完成时间（ISO 格式）
- 完成摘要（一句话）
```

### 5.3 文件路径

`docs/DevLog-Global.md`

### 5.4 生成逻辑

- 每次**首次完成归档**（`isCompletion: true`）时自动追加到全局日志
- 后续改进迭代**不重复记录**到全局日志
- 按时间倒序排列

| 项目 | 路径 |
|------|------|
| **核心引擎** | `lib/devlog-engine.ts` |
| **测试文件** | `tests/devlog-engine.test.ts` |

### 6.4 版本号规范

遵循语义化版本 `MAJOR.MINOR.PATCH`：

- **MAJOR**: 模板结构重大变更，不向后兼容

- **PATCH**: 文案修正、格式微调

## 7. 执行约束

### ✅ 必须遵守

- **任务状态前置：** 自动触发前必须检查任务是否已完成，未完成则不生成笔记
- **手动触发无限制：** 用户主动发起的记录不受任务状态约束
- **去重校验：** 每次写入前检查内容是否重复，重复则拦截并提示
- **同目标聚合：** 同一目标的改进追加到已有笔记，不创建新条目
- **严禁废话：** 直接输出结构化笔记，不输出寒暄

### ⚠️ 注意事项

- **上下文裁剪：** 如果对话历史较长，优先检索最近 3-5 轮与目标相关的讨论
- **重复归档：** 同一目标在同一 session 内只归档一次，避免重复
- **确认机制：** 生成笔记后询问用户"是否追加到 DevLog.md"，用户确认后执行写入

### ❌ 禁止行为

- 不修改 DevLog.md 中已有的历史记录
- 不自动删除或覆盖任何用户已确认的内容
- 不对未完成的目标自动归档（手动触发除外）

## 8. 快速参考

### 触发逻辑速查表

| 场景 | 任务状态 | 触发方式 | 是否生成笔记 |
|------|---------|---------|:-----------:|
| 完成任务后切换话题 | completed | 自动 | ✅ |
| 中途跳转到其他话题 | in_progress | 自动 | ❌ 静默跳过 |
| 用户说"记录一下" | 任意 | 手动 | ✅ |
| 用户说"搞定了" | → completed | 标记状态 | 待切换时触发 |
| 同一目标改进 | 任意 | 手动/自动 | ✅ 追加到已有 |
