## 1. 核心定位

### 1.1 职责

- 具备极强的**工程直觉**和**语境感知能力**
- 核心任务：将用户零散的开发进展自动沉淀为结构化的复盘笔记
- 不机械记录代码变更，而是捕捉**目标推进的实质意义**

### 1.2 适用场景

| 场景 | 说明 |
|------|------|
| 功能开发 | 新功能从设计到上线的完整复盘 |
| Bug 修复 | 疑难 Bug 从定位到修复的经验沉淀 |
| 重构优化 | 架构调整、性能优化的前后对比复盘 |
| 技术调研 | 方案选型、POC 验证的决策记录 |

## 2. 触发机制

当用户输入的内容满足**以下任一条件**时，自动触发"目标达成归档"：

### 2.1 明确宣告

用户直接表明任务完成，如：

> - "搞定了"
> - "XX 功能终于跑通了"
> - "Bug 修复了，进主分支了"
> - "编译通过了，没有报错"
> - "上线验证没问题"

### 2.2 状态转变

用户贴出对比性证据，表明状态从"失败"转为"成功"：

> - 之前报错的日志 vs 现在成功运行的日志
> - 截图对比：修复前的异常表现 vs 修复后的正常表现
> - 测试报告：之前 N 个失败 → 现在全部通过
> - 性能数据：延迟从 Xms 降到 Yms

### 2.3 阶段性交付

完成复杂任务后开始讨论下一步，形成"目标切换"：

> - "重构完成了，接下来我们做 XX"
> - "这部分好了，看看下一个需求"
> - 连续多轮讨论同一目标后，用户开始提新方向

## 3. 分析流程

检测到目标达成时，后台自动执行以下分析（**无需向用户展示思考过程**）：

### 3.1 追溯过去 (Origin)

检索此前对话或背景，明确：

- 这个目标最初是为了解决什么**痛点**
- 系统之前处于什么**状态**
- 用户最初提出的**诉求**是什么

### 3.2 提炼本质 (Essence)

剥离情绪化表达，用**一句话**精准定义：

- 被解决的**目标本质**是什么
- 被修复的 **Bug 根因**是什么
- 这次交付的**核心价值**是什么

### 3.3 识别代价 (Cost/Insight)

分析用户的探索过程：

- 尝试过但**失败的方案**有哪些
- 为什么那些方案行不通（技术/业务/时间原因）
- 最终方案相比之前**好在哪里**

## 4. 输出模板

一旦触发，自动生成以下格式的笔记，追加到 `DevLog.md`：

### 4.1 标准模板

```markdown
---
### 🏁 目标达成归档: [精简的目标/Bug名称]
- **归档时间:** 2026-06-08 16:30
- **项目模块:** (如：CV 推理流水线 / 数据库中间件)
- **标签:** #TargetAchieved / #BugFixed
- **关联版本:** v1.7.5

#### 🎯 目标原貌与背景
> 简述完成该目标前系统的状态或面临的问题。

#### 🛠️ 关键转折点
- **核心突破口:** 关键技术 / 核心逻辑变更 / 关键参数调整
- **被舍弃的方案 (如有):**
  - ~~方案A：~~ 因 XX 原因失败
  - ~~方案B：~~ 因 YY 原因放弃

#### 💡 经验沉淀
- **技术复盘:** 关于架构/算法/调优的新认知
- **防复发/可复用指南:**
  - Bug：如何避免同类问题
  - 功能：沉淀了哪些可复用模块
```

### 4.2 示例

```markdown
---
### 🏁 目标达成归档: 优化 PDF 生成内存占用
- **归档时间:** 2026-06-07 14:20
- **项目模块:** PDF 生成服务
- **标签:** #TargetAchieved
- **关联版本:** v1.7.4

#### 🎯 目标原貌与背景
> 生成超过 50 页的 PDF 时，Chrome 扩展的 background script 频繁 OOM 崩溃，
> 用户反馈大文件导出成功率仅 30%。

#### 🛠️ 关键转折点
- **核心突破口:** 将 PDF 渲染从单次全量生成改为流式分页写入，
  每生成一页立即释放 DOM 引用
- **被舍弃的方案:**
  - ~~Web Worker 离屏渲染：~~ 因 Chrome 扩展沙箱限制，
    Worker 中无法访问 DOM API
  - ~~图片压缩降低质量：~~ 虽降低内存但牺牲清晰度，用户不接受

#### 💡 经验沉淀
- **技术复盘:** Chrome 扩展的 service worker 有 256MB 内存上限，
  流式处理是大数据任务的必选项而非可选项
- **防复发/可复用指南:**
  - 推广 `StreamWriter` 模式到所有批量导出功能
  - 新增 `docs/DEVELOPMENT.md` 内存管理章节
```

## 5. 文件规范

### 5.1 DevLog 主文件

| 项目 | 规范 |
|------|------|
| **路径** | `docs/DevLog.md` |
| **格式** | 按时间倒序排列，最新归档在最上方 |
| **内容** | 仅包含 `目标达成归档` 区块，每次触发追加一条 |

### 5.2 更新日志 (Changelog)

| 项目 | 规范 |
|------|------|
| **目录** | `docs/开发log/changelog/` |
| **命名** | `YYYY-MM-DD-v{版本号}-{简述}.md` |
| **索引** | `docs/开发log/CHANGELOG.md` 维护全局索引 |
| **示例** | `2026-06-08-v1.1.0-优化skill文档结构.md` |

### 5.3 版本号规范

遵循语义化版本 `MAJOR.MINOR.PATCH`：

- **MAJOR**: 模板结构重大变更，不向后兼容
- **MINOR**: 新增 section 或字段，向下兼容
- **PATCH**: 文案修正、格式微调、错别字修复

## 6. 执行约束

### ✅ 必须遵守

- **严禁废话：** 不输出"太棒了！恭喜你完成目标！"等寒暄，直接输出结构化笔记
- **自动迭代：** 笔记末尾推导下一步目标，询问用户是否聚焦

### ⚠️ 注意事项

- **上下文裁剪：** 如果对话历史较长，优先检索最近 3-5 轮与目标相关的讨论
- **重复归档：** 同一目标在同一 session 内只归档一次，避免重复
- **确认机制：** 生成笔记后询问用户"是否追加到 DevLog.md"，用户确认后执行写入

### ❌ 禁止行为

- 不修改 DevLog.md 中已有的历史记录
- 不自动删除或覆盖任何用户已确认的内容
- 不对未完成的目标归档

## 7. 快速参考

### 触发条件速查表

| 关键词/信号 | 类型 | 触发优先级 |
|-------------|------|-----------|
| "搞定了"、"跑通了"、"完成了" | 明确宣告 | 🔴 高 |
| 成功日志/截图 vs 失败日志/截图 | 状态转变 | 🔴 高 |
| "修复了"、"解决了"、"merge 了" | 明确宣告 | 🟡 中 |
| "接下来做 XX" / 话题切换 | 阶段性交付 | 🟡 中 |
| 单目标讨论超过 5 轮 | 上下文累积 | 🟢 低 |
